# 全局编排器 — 软件生命周期 Skill 调用链（AI友好版）

> 本文件是所有 lyspec-* skill 的**全局调度入口**。
> 支持两种模式：**A路径**（全新开发）和 **B路径**（需求变更）。
> A路径定义完整的 skill 调用链路、项目阶段判断、各环节跳过条件。
> B路径定义需求变更的提案→执行→归档流程，选择性调用现有 skill。

---

## 路径判断（首要步骤）

AI在与用户交互的**第一步**，判断走哪条路径：

```
用户输入
  │
  ├── 识别为"全新开发" → A路径（全新开发流程）
  │   关键词：开始项目、启动开发、新系统、新功能开发、从PRD开始
  │
  ├── 识别为"需求变更" → B路径（需求变更流程）
  │   关键词：需求变更、功能调整、需求修改、变更提案、改需求
  │
  └── 不明确 → 询问用户
      "您要做什么？
       A. 全新开发（新系统/大版本升级/新功能）
       B. 需求变更（对已有项目的需求调整）"
```

> **判断依据**：
> - 用户提到"已有系统"、"原来的功能"、"之前的PRD"、"修改"、"调整" → 倾向B路径
> - 用户提到"新项目"、"从头开始"、"新系统" → 倾向A路径
> - 有疑问时主动询问，不猜测

## A路径：全新开发流程

> A路径是原有的完整开发流程，**零改动**。

### 全局变量（一次确认，全链传递）

AI在首次与用户交互时确认以下全局变量，后续所有 skill 直接使用，不再重复询问：

| 变量 | 确认方式 | 说明 |
|------|---------|------|
| `projectStage` | 询问用户 | `new`（新系统/大版本升级） / `iteration`（小功能迭代） |
| `projectName` | 从PRD提取或用户指定 | 项目简称，2-6个汉字或英文，用于所有产物文件命名 |
| `projectVersion` | 从PRD 3.3版本规划提取 | 格式 `V{X.Y}`（如 V1.0），用于所有产物文件命名 |
| `outputFormat` | 询问用户 | `md`（仅Markdown） / `md+word`（Markdown + Word） |

### 项目阶段判断规则

| 条件 | 判定为 `new` | 判定为 `iteration` |
|------|-------------|-------------------|
| 首次开发 | ✅ | — |
| 技术栈变更 | ✅ | — |
| 系统重构 | ✅ | — |
| 新增子系统 | ✅ | — |
| 在已有系统上新增功能页面 | — | ✅ |
| 修改已有业务逻辑 | — | ✅ |
| 增加接口 | — | ✅ |

> AI须在开始时询问："这是新系统/大版本升级，还是在已有系统上的小功能迭代？"

### Skill 调用链路

```
用户输入需求
  │
  ├──→ [全局] 确认 projectStage + projectName + outputFormat
  │
  ├──→ ① lyspec-prd（始终执行）
  │         │  new → 完整PRD（9章）
  │         │  iteration → 增量更新模式（基于已有PRD调整）
  │         ↓ 产物：PRD文档
  │
  ├──→ ② lyspec-ued（条件执行）
  │         │  new → 默认执行
  │         │  iteration → 主动询问"是否需要UED设计？"
  │         ↓ 产物：HTML Demo + 页面清单（如执行）
  │
  ├──→ ③ lyspec-arch（条件执行）
  │         │  new → 必需执行
  │         │  iteration → 主动询问"是否需要架构设计？"
  │         ↓ 产物：架构设计文档（如执行）
  │
  ├──→ ④ lyspec-detail（始终执行）
  │         │  编排 lyspec-backend + lyspec-frontend
  │         │  后端先行，围绕接口编程
  │         ↓ 产物：详细设计文档 + DDL
  │
  ├──→ ⑤ [代码编写]（执行环节，无独立skill）
  │         │  基于详细设计产物编写代码
  │         ↓ 产物：源代码
  │
  ├──→ ⑥ lyspec-unit-test（★ 强制执行，不可跳过）
  │         │  必须生成单元测试并执行通过
  │         ↓ 产物：测试代码（全部通过）
  │
  ├──→ ⑦ lyspec-code-review（★ 强制执行，不可跳过）
  │         │  代码评审与质量门禁
  │         │  6大评审维度：设计一致性、编码规范、逻辑正确性、安全合规、性能隐患、可维护性
  │         │  评审中若修改代码，重新执行单元测试确保无回归
  │         ↓ 产物：评审报告 + 修复后的代码（评审通过后代码冻结）
  │
  ├──→ ⑧ lyspec-test-cases（始终执行）
  │         │  基于评审通过后的最终代码生成功能测试用例
  │         ↓ 产物：测试用例 JSON + Excel
  │
  └──→ ⑨ lyspec-deploy（可选执行）
            │  询问用户是否需要生成部署文档和脚本
            │  裸机部署（默认）/ Docker部署
            ↓ 产物：部署文档 + 部署脚本 + Nginx配置
```

## B路径：需求变更流程

> B路径用于已有项目的需求调整，通过变更控制三步曲完成。
> 不重走全链路，而是**选择性触发**受影响的环节。

### B路径调用链路

```
用户描述变更需求
  │
  ├──→ [B-1] lyspec-change-propose（变更提案）
  │         │  探索变更背景 → 影响分析 → 生成提案+任务清单
  │         ↓ 产物：change-proposal.md + change-tasks.md
  │         │
  │         [暂停] 用户确认变更提案
  │
  ├──→ [B-2] lyspec-change-apply（变更执行）
  │         │  按任务清单选择性调用现有 lyspec-* skill
  │         │  支持暂停/恢复，遇设计冲突可回退到提案修订
  │         ↓ 产物：更新后的各环节文档
  │         │
  │         [暂停] 所有任务完成
  │
  └──→ [B-3] lyspec-change-archive（变更归档）
            │  检查完成状态 → 同步PRD变更记录 → 归档变更目录
            ↓ 产物：归档记录 + 变更总结
```

### B路径与A路径的关系

| 维度 | A路径（全新开发） | B路径（需求变更） |
|------|-----------------|-----------------|
| 触发场景 | 新系统、大版本升级、新功能开发 | 已有项目的需求调整 |
| 执行范围 | 全链路顺序执行 | 仅受影响环节选择性执行 |
| 产物模式 | 全新生成 | 增量更新已有产物 |
| 调用的 skill | 按顺序调用 ①~⑨ | 按影响链调用受影响的 skill |
| 变更追溯 | 无（首次生成） | 有完整的变更提案→执行→归档记录 |

### B路径影响传播规则

> 变更执行时，按以下依赖关系判断哪些环节受影响：

| 变更类型 | 受影响的 Skill |
|---------|---------------|
| 新增功能页面 | PRD + UED + 详设(前端+后端) + 单测 + 代码评审 + 测试用例 |
| 修改业务规则 | PRD + 详设(后端) + 单测 + 代码评审 + 测试用例 |
| 新增/修改接口 | PRD + 详设(后端+前端) + 单测 + 代码评审 + 测试用例 |
| 纯UI调整 | UED + 详设(前端) |
| 新增外部集成 | PRD + 架构 + 详设(后端) + 单测 + 代码评审 + 测试用例 |
| 非功能需求变化 | 架构 + 详设 |
| 数据模型变更 | 详设(后端) + 单测 + 代码评审 + 测试用例 |
| 权限模型变更 | PRD + 详设(前端+后端) + 代码评审 + 测试用例 |

> 详细的影响分析由 `lyspec-change-propose` 执行。

## 产物目录结构与命名规范

### 目录结构

```
{项目根目录}/
├── doc/                              ← 交付物（需审阅、归档、交付给客户）
│   ├── {原始需求文档}                  ← 用户提供的输入文档（不改名）
│   ├── {projectName}_{projectVersion}_PRD_{日期}.md
│   ├── {projectName}_{projectVersion}_ARCH_{日期}.md
│   ├── {projectName}_{projectVersion}_DETAIL_{日期}.md
│   ├── {projectName}_{projectVersion}_DDL_{日期}.sql
│   ├── {projectName}_{projectVersion}_UED/
│   │   ├── README.md                 ← UI/UE 设计说明文档
│   │   ├── pages/*.html              ← HTML Demo
│   │   └── assets/
│   ├── {projectName}_{projectVersion}_TC_{日期}.json
│   ├── {projectName}_{projectVersion}_TC_{日期}.xlsx
│   ├── {projectName}_{projectVersion}_TR_{日期}.md
│   ├── {projectName}_{projectVersion}_DEPLOY_{日期}.md
│   └── changes/                      ← B路径变更归档目录
│
└── output/                           ← 工程中间物（可重新生成、不交付）
    ├── generate_*.py                 ← 生成脚本
    ├── run_*.py                      ← 执行脚本
    ├── intermediate/                 ← 中间结果
    └── *.json                        ← 运行时快照
```

> **分界原则**："删掉后需要人重新评审的"放 `doc/`，"删掉后跑一遍脚本就能恢复的"放 `output/`。

### 产物命名规范

**命名模板**：`{projectName}_{projectVersion}_{文档类型}_{日期}.{ext}`

| 字段 | 规则 | 示例 |
|------|------|------|
| `projectName` | 全局变量，项目简称，全项目统一 | `智能巡检` |
| `projectVersion` | 全局变量，从PRD 3.3取值 | `V1.0` |
| 文档类型 | 固定枚举（见下表） | `PRD` |
| 日期 | `YYYY-MM-DD` 格式 | `2026-03-12` |

**文档类型枚举**：

| 类型标识 | 对应 Skill | 产物说明 |
|---------|-----------|---------|
| `PRD` | lyspec-prd | 产品需求规格说明书 |
| `UED` | lyspec-ued | UED 目录（README.md + HTML Demo） |
| `ARCH` | lyspec-arch | 架构设计文档 |
| `DETAIL` | lyspec-detail | 详细设计文档 |
| `DDL` | lyspec-detail | 数据库建表脚本 |
| `TC` | lyspec-test-cases | 测试用例（JSON/Excel） |
| `TR` | lyspec-test-cases | 测试报告 |
| `DEPLOY` | lyspec-deploy | 部署文档 |

**多版本共存示例**：

```
doc/
├── 智能巡检_V1.0_PRD_2026-03-12.md      ← V1.0 版本
├── 智能巡检_V1.5_PRD_2026-05-20.md      ← V1.5 版本，互不冲突
├── 智能巡检_V1.0_ARCH_2026-03-12.md
├── 智能巡检_V1.0_UED/
└── 智能巡检_V1.5_UED/                   ← V1.5 的 UED 目录
```

> **AI执行规则**：
> - 所有 lyspec-* skill 输出产物时必须使用此命名规范，不得自行发明命名
> - `projectName` 和 `projectVersion` 在编排器首次交互时确认，全链传递
> - 同版本同日期更新直接覆盖（git 历史保留中间版本）
> - 用户指定了自定义文件名时，以用户指定为准

### 产物输出格式规则

| `outputFormat` | Markdown | Word | Excel |
|----------------|----------|------|-------|
| `md` | ✅ 所有文档 | — | 测试用例始终Excel |
| `md+word` | ✅ 所有文档 | ✅ PRD、架构设计、详细设计 | 测试用例始终Excel |

> Word 文件通过 `docx` skill 从 Markdown 转换生成。
> 测试用例（lyspec-test-cases）始终输出 JSON + Excel，不受 `outputFormat` 影响。
> UED产物（HTML Demo）不受 `outputFormat` 影响。

## 上下游数据流

```
① PRD
├──→ ② UED：用户角色、功能清单、用例交互
├──→ ③ 架构设计：系统边界、功能结构、集成需求、非功能需求
├──→ ④ 详细设计：功能清单、用例、数据实体
└──→ ⑧ 测试用例：用例正常/异常流、验收标准

② UED
└──→ ④ 详细设计：HTML Demo、页面清单、组件树

③ 架构设计
└──→ ④ 详细设计：模块划分、技术选型、非功能需求

④ 详细设计
├──→ ⑤ 代码编写：类图→代码骨架；接口→方法签名；DDL→Entity
├──→ ⑥ 单元测试：分层架构→测试策略；异常体系→异常路径覆盖
├──→ ⑦ 代码评审：接口设计→设计一致性基准；分层架构→职责检查
└──→ ⑧ 测试用例：接口清单→接口测试；错误码→异常场景

⑤ 代码
├──→ ⑥ 单元测试：被测源代码→测试目标（必须执行通过）
└──→ ⑦ 代码评审：源代码→评审目标（评审中修改代码须重跑⑥单测）

⑥ 单元测试
└──→ ⑦ 代码评审：覆盖率报告→测试充分性确认

⑦ 代码评审（评审通过后代码冻结）
└──→ ⑧ 测试用例：基于最终代码生成功能测试用例，与单测互补（代码级 vs 场景级）

⑨ 部署（可选，从上游提取信息）
├── ③ 架构设计：部署架构图、服务器规划、中间件清单
├── ④ 详细设计：技术选型、数据库类型、依赖服务
└── ⑤ 代码：项目结构、pom.xml/package.json、端口配置
```

## AI交互入口脚本

当用户触发全局编排器时，AI按以下顺序执行：

```
0. 判断路径：识别用户意图 → A路径（全新开发）或 B路径（需求变更）
   若不明确，询问："您要做什么？A.全新开发 B.需求变更"

【A路径】
1. 询问："这是新系统/大版本升级，还是小功能迭代？" → 锁定 projectStage
2. 询问："请提供项目名称" → 锁定 projectName
3. 询问："文档需要同时输出 Word 格式吗？（默认仅Markdown）" → 锁定 outputFormat
4. 询问："请提供需求背景材料（文档/口述均可）"
5. 按调用链路逐步执行，每个 skill 完成后提示用户：
   "✅ {skill名称} 已完成，下一步将执行 {下一个skill名称}，是否继续？"
6. 条件执行的 skill（UED、架构设计）在到达时询问用户是否跳过
7. 强制执行的 skill（单元测试、代码评审）不提供跳过选项
8. 可选执行的 skill（部署）在到达时询问："是否需要生成部署文档和脚本？"

【B路径】
1. 调用 lyspec-change-propose（变更提案）
   → 探索背景 → 影响分析 → 生成提案+任务清单 → 用户确认
2. 调用 lyspec-change-apply（变更执行）
   → 按任务清单选择性调用现有 skill → 逐任务完成
3. 调用 lyspec-change-archive（变更归档）
   → 完成度检查 → PRD变更记录同步 → 归档 → 输出总结
```
