# 产品需求规格说明书 Skill（AI友好版）

> 本文档处于软件生命周期的**起点**："**PRD**→UI/UE设计→架构设计→详细设计→代码编写→单元测试→功能测试"。
> PRD无上游文档依赖，所有信息来源于用户输入 + AI结构化引导。
> PRD定义"做什么"和"为什么做"，不涉及"怎么做"（实现细节由下游 skill 承接）。

---

## 定位与边界

### 与下游 skill 的分工

| 维度 | lyspec-prd（本 skill） | 下游 skill |
|------|------------------------|-----------|
| 关注层次 | 业务需求（What / Why） | 技术实现（How） |
| 功能描述粒度 | 用例级（用户目标、前置条件、主流程、异常流） | 接口级 / 组件级 / 类级 |
| 数据描述粒度 | 业务实体 + 核心属性 + 关联关系 | 数据库表结构 / ER图 / DDL |
| 接口描述粒度 | 系统与外部系统的集成关系（概念级） | RESTful API 设计 / 请求参数 / 响应体 |
| 非功能需求 | 业务视角（响应时间、并发量、数据量级） | 技术方案（缓存策略、分库分表、限流） |
| 产物格式 | Markdown 文档 | 设计文档 / 代码 / DDL / 测试用例 |

> **边界原则**：PRD 中不出现技术选型、数据库表名、API URL、类名等实现细节。

## 信息获取策略

> PRD 是生命周期起点，无上游文档可提取，所有内容来源于用户。

### 三级信息获取

| 级别 | 策略 | 说明 |
|------|------|------|
| ① 直接提取 | 用户已提供的文档/描述 | 用户给出的需求文档、会议纪要、竞品分析等 |
| ② AI推导 | 从已知信息合理推导 | 从功能描述推导异常流、从角色推导权限、从业务规则推导数据实体 |
| ③ 待用户确认 | 无法推导的关键决策 | 标注为"⚠️ 待确认"，不编造 |

> **AI生成规则**：
> - ①②类信息直接输出，标注来源（"来源：用户提供" / "来源：AI推导"）
> - ③类信息必须标注"⚠️ 待确认"，暂停等待用户补充
> - 严禁编造业务数据（如并发量、数据量、性能指标），必须由用户确认

## AI执行流程

### 前置：收集需求背景

1. **确认输入来源**：
   - "请提供需求背景材料（需求文档、会议纪要、竞品分析等均可）"
   - "如果没有现成文档，请口述核心需求，我来帮你结构化"

2. **确认项目类型**：
   - 新产品 / 已有产品新模块 / 已有功能改进（小功能迭代）
   - 据此调整文档深度（新产品需完整9章；功能改进可精简引言和项目概述）

3. **确认是否有已有PRD**（功能改进/小迭代时）：
   - "是否有已有的PRD文档？如有请提供路径"
   - 若有已有PRD → 进入**增量更新模式**：基于已有PRD调整，生成新版本（文件名加日期区分版本）
   - 若无已有PRD → 进入**文档生成模式**：按完整流程执行

4. **确认文档范围**：
   - "本次PRD覆盖哪些功能模块？"
   - "是否需要包含非功能需求（性能、安全、兼容性）？"

## 文档章节结构

### 第一章：引言

| 节 | 内容 | 信息来源 |
|----|------|---------|
| 1.1 文档目的 | 本文档的目标读者和用途 | AI生成模板 + 用户确认 |
| 1.2 项目背景 | 项目发起原因、业务痛点、期望目标 | 用户提供 |
| 1.3 术语定义 | 业务领域术语及其含义 | 用户提供 + AI从上下文提取 |
| 1.4 参考文档 | 相关的规范、标准、竞品分析等 | 用户提供 |

### 第二章：项目概述

| 节 | 内容 | 信息来源 |
|----|------|---------|
| 2.1 产品愿景 | 一句话描述产品核心价值 | 用户提供 |
| 2.2 目标用户 | 用户角色清单（角色名、职责、使用场景） | 用户提供 + AI推导 |
| 2.3 系统边界 | 系统范围内/范围外的功能划分 | 用户提供 + AI推导 |
| 2.4 核心业务流程 | 端到端的业务流程图（Mermaid） | AI从功能描述推导，用户确认 |

> **AI生成规则**：
> - 2.2 用户角色必须输出为表格：角色名 | 职责描述 | 典型使用场景 | 权限概述
> - 2.3 系统边界必须明确列出"本系统负责" vs "本系统不负责"
> - 2.4 业务流程图使用 Mermaid flowchart 格式

### 第三章：功能总览

| 节 | 内容 | 信息来源 |
|----|------|---------|
| 3.1 功能结构图 | 树形功能模块分解（Mermaid） | AI从需求描述推导 |
| 3.2 功能清单 | 全量功能点列表（模块→功能→子功能） | AI结构化 + 用户确认 |
| 3.3 版本规划 | 各版本包含的功能范围（MVP / V1.0 / V2.0） | 用户提供 |
| 3.4 页面导航关系图 | 页面间跳转/跳出路径（Mermaid） | AI从第四章用例推导 |

> **功能清单表格模板**：

| 模块 | 功能 | 子功能 | 优先级 | 版本 | 说明 |
|------|------|--------|--------|------|------|
| 用户管理 | 用户注册 | 手机号注册 | P0 | V1.0 | — |
| 用户管理 | 用户注册 | 邮箱注册 | P1 | V1.5 | — |

> **AI生成规则**：
> - 3.1 功能结构图使用 Mermaid graph TD 格式，不超过3层
> - 3.2 功能清单须覆盖用户提到的所有功能点，优先级须与用户确认
> - 3.3 版本规划若用户未提供，标注"⚠️ 待确认"
> - 3.4 页面导航关系图须从第四章用例的主流程中提取所有"用户跳转/查看/进入"动作，使用 Mermaid graph LR 格式
>
> **功能清单颗粒度规则**（强制）：
>
> 每个「子功能」行必须满足以下条件之一，否则须继续拆分：
> 1. 可映射为一个独立页面（或一个独立覆盖层）
> 2. 可用一个用例（UC）完整描述其交互流程
>
> **拆分触发条件**（检测到以下情况时必须拆分）：
> - 描述中含逗号分隔的多个名词（如"报警阈值、推送策略、调度参数"→ 拆为 3 行子功能）
> - 描述中含"等"字暗示未穷举（如"配置 XX 等"→ 展开穷举所有配置类别）
> - 描述可对应多个 Tab 页签或多个独立表单区域
>
> 示例：
> ```
> ✗ 错误（笼统）: 系统参数配置 | 全局参数配置 | "报警阈值、推送策略、调度参数等"
> ✓ 正确（拆分）:
>   系统参数配置 | 基础参数配置 | 系统名称、巡检间隔、心跳间隔等基础运行参数
>   系统参数配置 | 报警参数配置 | 各等级确认时限、默认报警等级、升级策略
>   系统参数配置 | 调度参数配置 | 最小执行间隔、并发任务上限、重试策略
>   系统参数配置 | 推送参数配置 | 短信网关、通知模板、推送频率限制
> ```

> **页面导航关系图模板**：
>
> ```mermaid
> graph LR
>     A[订单列表] -->|点击"详情"| B[订单详情]
>     B -->|点击"物流跟踪"| C[物流详情]
>     B -->|点击"退款"| D[退款申请表单]
>     D -->|提交成功| E[退款进度页]
> ```
>
> AI生成规则：
> - 从第四章每个用例的主流程中提取页面跳转关系
> - 包含列表→详情、详情→关联页面、操作→结果页等所有跳转路径
> - 弹窗/抽屉类交互不在此图中体现（在用例的关键交互动作表中定义）

### 第四章：用例描述

> 这是PRD的核心章节，详细描述每个功能的用户交互流程。

#### 用例描述模板

每个核心功能按以下结构描述：

```
### UC-{编号}：{用例名称}

**参与角色**：{角色名}
**前置条件**：{触发该用例前须满足的条件}
**后置条件**：{用例完成后系统的状态变化}

**主流程（正常路径）**：
1. 用户执行操作A
2. 系统响应行为B
3. ...
4. 系统显示结果

**异常流程**：
- E1：{异常条件} → {系统响应}
- E2：{异常条件} → {系统响应}

**业务规则**：
- BR1：{规则描述}
- BR2：{规则描述}

**关键交互动作**：

| 动作触发 | 交互形态 | 包含信息/字段 | 校验/约束 | 结果反馈 |
|---------|---------|-------------|----------|---------|
| 点击"新建" | Modal弹窗 | 名称(必填)、类型(单选)、描述(选填) | 名称唯一校验 | 成功→刷新列表+Toast提示 |
| 点击"删除" | 确认弹窗 | "确定删除{名称}？此操作不可撤销" | 有关联引用时禁止删除 | 成功→移除行+Toast提示 |
| 点击行"详情" | 页面跳转 | 展示完整信息+关联数据+操作按钮 | — | — |

**状态-操作矩阵**（仅当该用例涉及状态流转时必填）：

| 当前状态 | 可执行操作 | 操作角色 | 目标状态 |
|---------|----------|---------|---------|
| 待审核 | 审核通过 / 审核退回 | 审核人 | 已通过 / 已退回 |
| 已退回 | 重新提交 | 提交人 | 待审核 |

**UI要求**（概念级）：
- 关键页面/交互说明（不含视觉设计细节，视觉设计由 lyspec-ued 承接）
```

> **AI生成规则**：
> - 每个用例必须包含至少一条异常流程
> - 异常流程从以下维度推导：输入校验失败、权限不足、数据冲突、外部依赖异常
> - 业务规则须可验证（下游 lyspec-test-cases 可据此生成测试用例）
> - UI要求仅描述页面信息要素和交互逻辑，不涉及颜色、字体、布局等视觉设计
> - **关键交互动作表**是本用例模板的**必填节**，每个用例至少包含3行（须覆盖新增/编辑/删除/查看详情中的至少3种操作）
> - 交互形态必须明确为以下之一：**Modal弹窗**（≤6个字段的新增/编辑/确认）、**Drawer抽屉**（>6个字段或需要参考上下文的表单）、**页面跳转**（独立的详情/编辑页）、**确认弹窗**（删除/审批等二次确认）、**Toast提示**（轻量反馈）
> - "包含信息/字段"列须列出弹窗/抽屉内的全部表单字段，并标注必填(必填)/选填
> - "校验/约束"列须从业务规则(BR)中提取对应的校验逻辑
> - **状态-操作矩阵**：凡第五章定义了状态机的业务实体，在对应用例中**必须**输出此矩阵，覆盖每个状态×每个角色的操作可见性；此矩阵直接作为 lyspec-ued 生成按钮显隐逻辑的依据

#### 用例变体模板

> 上方的通用用例模板适用于 **CRUD 型功能**（约 60% 的页面）。
> 以下 3 种变体模板适用于非 CRUD 功能，AI 须根据功能类型自动选择合适的变体。

**变体选择规则**：
- 功能清单子功能含"配置""参数""设置"关键词 → 使用 **变体 A（配置型）**
- 功能清单子功能含"大屏""监控""统计""分析""总览""看板"关键词 → 使用 **变体 B（展示型）**
- 功能清单子功能含"导入""导出""批量""上传""下载"关键词 → 使用 **变体 C（导入导出型）**
- 其他 → 使用上方通用 CRUD 用例模板

##### 变体 A：配置型用例

> 适用于系统参数配置、规则配置、通知设置等 Settings 型功能。
> 替代通用模板中的「关键交互动作表」，输出「配置项清单表」。

```
### UC-{编号}：{用例名称}（配置型）

**参与角色**：{角色名}
**前置条件**：{触发条件}
**后置条件**：配置已保存并生效

**主流程（正常路径）**：
1. 用户进入{配置}页面
2. 系统展示当前配置值（分 Tab 页签展示各配置分组）
3. 用户切换至目标 Tab，修改配置项
4. 用户点击"保存"
5. 系统校验配置值合法性并保存
6. 系统提示"配置保存成功"

**异常流程**：
- E1：{配置值超出范围} → 系统提示"{配置项}取值须在{范围}内"
- E2：{必填项为空} → 系统提示"{配置项}不能为空"

**配置项清单**：

| 配置分组 | 配置项名称 | 输入类型 | 默认值 | 取值范围/选项 | 校验规则 | 说明 |
|---------|----------|---------|-------|-------------|---------|------|
| {分组1} | {项名} | 数值/文本/单选/多选/开关 | {值} | {范围} | {规则} | {说明} |

**业务规则**：
- BR1：{规则}
```

> **AI生成规则（变体 A）**：
> - 每个配置分组在 UED 中映射为一个 Tab 页签，因此**每个分组至少包含 3 个配置项**
> - 输入类型必须明确为：数值(含单位) / 文本 / 单选(列出选项) / 多选(列出选项) / 开关(是/否) / 密码
> - 默认值必须填写（用户未提供时 AI 推导并标注"AI推导"）
> - 取值范围必须明确（数值型给出 min~max，文本型给出长度限制）
> - 此表直接作为 lyspec-ued 生成 Settings 页面各 Tab 表单的输入源

##### 变体 B：展示型用例

> 适用于看板、统计分析、实时监控等只读展示页面。
> 替代通用模板中的「关键交互动作表」，输出「展示要素清单表」。

```
### UC-{编号}：{用例名称}（展示型）

**参与角色**：{角色名}
**前置条件**：{数据来源已对接}
**后置条件**：无状态变更（只读展示）

**主流程（正常路径）**：
1. 用户进入{展示}页面
2. 系统加载并展示各区域数据
3. 用户可通过筛选条件（时间范围/区域等）调整展示内容
4. 用户可点击指标卡/图表/列表中的元素查看详情

**展示要素清单**：

| 展示区域 | 展示类型 | 数据内容 | 数据来源 | 刷新策略 | 交互动作 |
|---------|---------|---------|---------|---------|---------|
| 顶部指标行 | 指标卡×N | {各指标名+单位} | {业务实体.字段} | 实时/定时/手动刷新 | 点击→弹窗详情/无 |
| 左侧区域 | 折线图/柱状图/饼图 | {X轴含义} × {Y轴含义} | {数据来源} | {刷新策略} | 时间范围切换/无 |
| 右侧区域 | 滚动列表/排名列表 | {列表项内容} | {数据来源} | {刷新策略} | 点击→跳转详情/无 |
| 底部区域 | 数据表格 | {各列名称} | {数据来源} | {刷新策略} | 点击行→跳转/无 |

**筛选条件**：
| 筛选项 | 输入类型 | 选项/范围 | 默认值 |
|--------|---------|----------|-------|
| 时间范围 | 单选 | 近7天/近30天/近90天 | 近7天 |
```

> **AI生成规则（变体 B）**：
> - 展示类型必须明确为：指标卡 / 折线图 / 柱状图 / 饼图 / 表格 / 排名列表 / 滚动列表 / 地图
> - 每个指标卡的数据来源必须明确到业务实体和字段级别
> - 图表须标注 X 轴和 Y 轴含义（如"X=日期, Y=报警数量"）
> - 交互动作列须明确每个可点击元素的目标（跳转页面/弹窗/无交互）
> - 此表直接作为 lyspec-ued 确定页面布局和组件选择的输入源

##### 变体 C：导入导出型用例

> 适用于批量导入、报告导出、数据迁移等文件操作功能。
> 可作为 CRUD 型用例的补充节（附加在关键交互动作表之后）。

```
**导入导出规格**：

| 操作类型 | 触发方式 | 文件格式 | 包含字段 | 校验规则 | 错误处理 | 结果反馈 |
|---------|---------|---------|---------|---------|---------|---------|
| 批量导入 | Modal弹窗(上传区+模板下载) | Excel(.xlsx) | {字段列表} | {各字段校验} | 跳过错误行并汇总 | 成功N条/失败M条+错误明细 |
| 数据导出 | 按钮点击 | Excel/PDF | {导出字段} | — | — | 触发下载+Toast提示 |
```

> **AI生成规则（变体 C）**：
> - 导入操作必须描述模板格式（字段列表和示例值）
> - 导入错误处理策略必须明确（跳过/中止/回滚）
> - 导出操作必须明确导出范围（当前筛选结果/全量/选中项）
> - 此表可附加在 CRUD 型用例的关键交互动作表之后，不需要独立用例

### 第五章：数据需求

| 节 | 内容 | 信息来源 |
|----|------|---------|
| 5.1 核心业务实体 | 实体名、核心属性、实体间关系 | AI从用例推导 |
| 5.2 数据字典 | 枚举值、状态机、编码规则 | 用户提供 + AI推导 |
| 5.3 数据生命周期 | 数据的创建、更新、归档、删除规则 | 用户提供 |

> **AI生成规则**：
> - 5.1 实体关系使用 Mermaid erDiagram 格式，仅列业务属性，不涉及数据库字段设计
> - 5.2 状态机使用 Mermaid stateDiagram 格式
> - 实体属性描述格式：属性名 | 业务含义 | 数据类型（业务级：文本/数值/日期/枚举）| 是否必填 | 约束说明

## 分阶段输出策略

### 阶段划分

| 阶段 | 输出内容 | 暂停点 |
|------|----------|--------|
| 阶段一 | 第一~二章（引言 + 项目概述） | 确认背景理解、用户角色、系统边界 |
| 阶段二 | 第三~四章（功能总览 + 用例描述） | ★ 确认功能清单和核心用例（PRD核心内容） |
| 阶段三 | 第五~九章（数据 + 接口 + 非功能 + 约束 + 验收） | 确认完整性，输出最终文档 |

### 用户交互路线图

```
前置 ──→ [询问] 收集需求背景材料 + 确认项目类型 + 文档范围
  │
阶段一 ──→ [暂停] 确认引言 + 项目概述（角色、边界、业务流程）
  │
阶段二 ──→ [暂停] ★ 确认功能清单 + 核心用例描述（主流程、异常流、业务规则）
  │
阶段三 ──→ [暂停] 确认数据需求 + 接口需求 + 非功能 + 约束 + 验收标准
  │
完成 ──→ 输出完整PRD文档 + 下游衔接建议
```

> **阶段二说明**：这是PRD的核心确认点。功能清单和用例描述一旦确认，将作为所有下游 skill 的输入源头。
> AI须在此阶段充分引导用户补充异常流和业务规则，而非跳过进入下一阶段。
