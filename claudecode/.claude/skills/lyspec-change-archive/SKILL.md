# 需求变更归档 Skill（AI友好版）

> 本文档处于 LYSpec 变更控制流程的**终点**："变更提案→变更执行→**变更归档**"。
> 变更归档负责收尾工作：检查完成状态、同步增量到主文档、归档变更目录。

---

## 定位与职责

| 职责 | 说明 |
|------|------|
| 完成度检查 | 检查 change-tasks.md 中所有任务是否完成 |
| 主文档同步 | 将变更信息同步到 PRD 的变更记录章节 |
| 目录归档 | 将变更目录从 `output/changes/` 移至 `output/changes/archive/` |
| 变更总结 | 输出变更总结报告 |

---

## AI执行流程

### 第一步：选择变更

**与 lyspec-change-apply 相同的选择逻辑**：

- 若用户指定了变更名称 → 直接使用
- 若对话上下文中可推断 → 自动选中
- 若仅存在一个活跃变更 → 自动选中
- 若存在多个活跃变更 → 列出可用变更，询问用户选择
- 仅展示活跃变更（未归档的），已归档的不展示

> **重要**：不要在存在歧义时自动猜测，始终由用户确认。

---

### 第二步：完成度检查

#### 2.1 检查任务完成状态

读取 `output/changes/<changeName>/change-tasks.md`，统计任务状态：

```
统计结果：
- 总任务数：{M}
- 已完成：{N}（标记为 [x] 的）
- 未完成：{M-N}（标记为 [ ] 的）
```

**若全部完成**：继续下一步。

**若存在未完成任务**：

```
⚠️ 警告：以下任务尚未完成

- [ ] ⑥ 单元测试更新 — 补充会签逻辑测试
- [ ] ⑦ 测试用例更新 — 新增会签流程测试场景

是否仍要归档？
1. 先完成剩余任务（推荐）→ 执行 lyspec-change-apply 继续
2. 仍然归档（标注为部分完成）
3. 取消归档
```

> 不因未完成而阻止归档，但必须告知用户并获得确认。

#### 2.2 检查产物一致性

快速校验变更执行后的产物：

| 检查项 | 方法 | 处理 |
|--------|------|------|
| 变更提案文件存在 | 检查 change-proposal.md | 不存在则警告 |
| 任务清单文件存在 | 检查 change-tasks.md | 不存在则警告 |
| 已完成任务有完成摘要 | 检查 [x] 项是否有 ✅ 完成记录 | 缺失则提醒 |

---

### 第三步：主文档同步

> 将变更信息同步回主文档，确保主文档反映最新状态。

#### 3.1 PRD变更记录同步

若 PRD 存在，在 PRD 文档末尾追加变更记录：

```markdown
## 变更记录

| 版本 | 日期 | 变更名称 | 变更内容 | 影响范围 |
|------|------|---------|---------|---------|
| V1.1 | {YYYY-MM-DD} | {changeName} | {变更摘要} | {受影响环节列表} |
```

> **规则**：
> - 若 PRD 已有"变更记录"章节，追加新行
> - 若 PRD 无"变更记录"章节，在文档末尾新增该章节
> - 版本号基于已有版本递增（V1.0 → V1.1 → V1.2）
> - 变更摘要从 change-proposal.md 的"变更描述"提取，精简为一句话

#### 3.2 其他产物同步说明

| 产物 | 同步方式 | 说明 |
|------|---------|------|
| PRD | 追加变更记录章节 | 本步骤自动执行 |
| 详细设计 | 已在 change-apply 阶段更新 | 无需额外同步 |
| 测试用例 | 已在 change-apply 阶段更新 | 无需额外同步 |
| 架构设计 | 已在 change-apply 阶段更新（如涉及） | 无需额外同步 |

> 除 PRD 变更记录外，其他产物在 `lyspec-change-apply` 执行阶段已完成增量更新，归档阶段不再修改。

---

### 第四步：执行归档

#### 4.1 创建归档目录

```bash
mkdir -p output/changes/archive
```

#### 4.2 生成归档目录名

格式：`YYYY-MM-DD-<changeName>`

示例：`2026-03-11-add-export-feature`

#### 4.3 检查目标是否已存在

- 若 `output/changes/archive/YYYY-MM-DD-<changeName>` 已存在：
  - 报错并建议使用带序号的名称：`YYYY-MM-DD-<changeName>-2`
- 若不存在：继续

#### 4.4 移动变更目录

```bash
mv output/changes/<changeName> output/changes/archive/YYYY-MM-DD-<changeName>
```

归档后的目录结构：

```
output/changes/
├── archive/
│   └── 2026-03-11-add-export-feature/
│       ├── change-proposal.md
│       └── change-tasks.md
└── modify-approval-flow/          ← 其他活跃变更不受影响
```

---

### 第五步：输出变更总结

```
## 变更归档完成

**变更名称**：{changeName}
**提案日期**：{proposalDate}
**归档日期**：{archiveDate}
**归档位置**：output/changes/archive/{YYYY-MM-DD-changeName}/

### 变更摘要
{从 change-proposal.md 提取的变更描述}

### 完成状态
| 任务 | 状态 |
|------|------|
| ① PRD增量更新 | ✅ 已完成 |
| ④ 详细设计更新 | ✅ 已完成 |
| ⑥ 单元测试更新 | ✅ 已完成 |
| ⑦ 测试用例更新 | ✅ 已完成 |

### 更新的产物
| 产物 | 文件路径 |
|------|---------|
| PRD（含变更记录） | output/{项目名}_PRD_{日期}.md |
| 详细设计 | output/{模块名}_详细设计_{日期}.md |
| 测试用例 | output/{模块名}_test_cases.json |

### PRD版本变化
V1.0 → V1.1（新增变更记录章节）

{若有未完成任务}
### ⚠️ 注意
以下任务在归档时尚未完成：
- [ ] ⑥ 单元测试更新
建议在后续迭代中补充完成。
```

## 特殊场景处理

### 变更目录不存在

若 `output/changes/<changeName>` 不存在：
- 检查归档目录中是否已有同名变更
- 已归档 → 告知用户："该变更已于 {日期} 归档至 {路径}"
- 均不存在 → 告知用户："未找到该变更，请先执行 lyspec-change-propose 创建变更提案"

### 重复归档

若用户尝试归档一个已归档的变更：
- 告知用户："该变更已归档至 {路径}，无需重复归档"

### 归档后需要回滚

若用户归档后发现需要修改：
- 建议新建一个变更（如 `fix-export-feature`），而非从归档中恢复
- 原因：归档是不可变记录，保持变更历史的完整性
