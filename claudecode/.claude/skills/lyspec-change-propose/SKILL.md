# 需求变更提案 Skill（AI友好版）

> 本文档处于 LYSpec 变更控制流程的**起点**："**变更提案**→变更执行→变更归档"。
> 变更提案定义"改什么"、"为什么改"、"影响哪些环节"，不涉及具体实现。
> 变更提案完成后，由 `lyspec-change-apply` 接管执行。

---

## 定位与边界

### 与全新开发流程的区别

| 维度 | lyspec-orchestrator A路径（全新开发） | lyspec-change-propose（本 skill） |
|------|---------------------------------------|-----------------------------------|
| 触发场景 | 新系统、大版本升级、新功能开发 | 已有项目的需求调整、功能修改、业务规则变化 |
| 执行范围 | 全链路顺序执行（PRD→UED→架构→详设→...） | 仅影响链上的环节选择性执行 |
| 产物类型 | 全新文档 | 增量更新 + 变更记录 |
| 入口 | "开始项目"、"启动开发流程" | "需求变更"、"变更提案"、"功能调整" |

### 与其他变更 skill 的分工

| Skill | 职责 | 产物 |
|-------|------|------|
| **lyspec-change-propose**（本 skill） | 探索背景 + 影响分析 + 生成提案 | change-proposal.md + change-tasks.md |
| lyspec-change-apply | 按任务清单选择性调用现有 skill | 更新后的各环节文档 |
| lyspec-change-archive | 归档变更 + 同步回主文档 | 归档目录 + 变更总结 |

---

## 全局变量

AI在首次交互时确认以下变量：

| 变量 | 确认方式 | 说明 |
|------|---------|------|
| `projectName` | 从已有产物提取或用户指定 | 当前项目名称 |
| `changeName` | 用户描述 → AI推导 kebab-case | 变更标识（如 `add-export-feature`、`modify-approval-flow`） |
| `existingArtifacts` | AI扫描 output/ 目录 | 已有的产物清单（PRD、UED、架构、详设、测试用例等） |

## AI执行流程

### 第一步：探索变更背景

> 这是变更提案的"想清楚"环节，对应 OpenSpec explore 的思路，但不是独立 skill。

**1.1 确认变更来源**

询问用户：
- "请描述本次需求变更的内容（口述、文档、截图均可）"
- "变更的原因是什么？（客户反馈/业务调整/技术限制/...）"

**1.2 扫描已有产物**

自动扫描 `output/` 目录，识别已有产物：

```
output/
├── {项目名}_PRD_{日期}.md          → 已有PRD
├── {需求名}-ai-ued-detail/         → 已有UED产物
├── {项目名}_架构设计_{日期}.md      → 已有架构设计
├── {模块名}_详细设计_{日期}.md      → 已有详细设计
├── {模块名}_test_cases.json        → 已有测试用例
└── changes/                        → 已有变更记录
```

> 若 output/ 目录不存在或为空，提示用户："未找到已有产物，建议通过 lyspec-orchestrator A路径（全新开发）执行。"

**1.3 阅读相关产物**

根据用户描述的变更范围，读取相关的已有产物：
- 若变更涉及功能调整 → 读取 PRD 的功能清单（第三章）和用例描述（第四章）
- 若变更涉及界面调整 → 读取 UED 的页面清单和组件树
- 若变更涉及接口调整 → 读取详细设计的接口清单
- 若变更涉及数据模型调整 → 读取详细设计的数据设计章节

**1.4 与用户对话澄清**

- 主动提问澄清模糊点，一次只问一个问题
- 必要时用 ASCII 图、对比表辅助分析
- 引用已有产物中的具体内容帮助用户定位变更范围

**示例对话**：

```
用户：审批流程需要加一个会签节点

AI：我已读取现有PRD（第四章 UC-005：审批流程），当前流程为：

    提交 → 部门审批 → 总经理审批 → 完成

    你期望的会签节点加在哪个位置？

    方案A：提交 → 部门审批 → [会签] → 总经理审批 → 完成
    方案B：提交 → [会签] → 部门审批 → 总经理审批 → 完成
    方案C：其他位置
```

> **约束**：探索环节只读文件、做分析，不写代码、不修改任何产物。

**探索环节完成标志**：AI能清晰回答以下问题：
- 改什么？（变更的具体内容）
- 为什么改？（变更原因）
- 改多大？（变更范围边界）

### 第二步：影响分析

> 基于 lyspec-orchestrator 定义的上下游数据流，自动推导变更影响链。

#### 影响传播规则

根据 LYSpec 的上下游依赖关系：

```
① PRD
├──→ ② UED：用户角色、功能清单、用例交互
├──→ ③ 架构设计：系统边界、功能结构、集成需求、非功能需求
├──→ ④ 详细设计：功能清单、用例、数据实体
└──→ ⑦ 测试用例：用例正常/异常流、验收标准

② UED
└──→ ④ 详细设计：HTML Demo、页面清单、组件树

③ 架构设计
└──→ ④ 详细设计：模块划分、技术选型、非功能需求

④ 详细设计
├──→ ⑤ 代码编写：类图→代码骨架；接口→方法签名；DDL→Entity
├──→ ⑥ 单元测试：分层架构→测试策略；异常体系→异常路径覆盖
└──→ ⑦ 测试用例：接口清单→接口测试；错误码→异常场景
```

#### 变更类型与影响映射

| 变更类型 | 影响的 Skill | 影响判定依据 |
|---------|-------------|-------------|
| 新增功能页面 | PRD + UED + 详设(前端+后端) + 单测 + 测试用例 | PRD功能清单新增条目 |
| 修改业务规则 | PRD + 详设(后端) + 单测 + 测试用例 | PRD用例的业务规则变化 |
| 修改审批/流程 | PRD + 详设(后端) + 单测 + 测试用例 | PRD用例的主流程/异常流变化 |
| 新增接口 | PRD + 详设(后端) + 单测 + 测试用例 | 详设接口清单新增条目 |
| 修改接口参数 | 详设(后端+前端) + 单测 + 测试用例 | 详设接口参数表变化 |
| 纯UI调整 | UED + 详设(前端) | UED页面/组件变化，不涉及后端 |
| 新增外部集成 | PRD + 架构 + 详设(后端) + 单测 + 测试用例 | PRD外部集成章节新增 |
| 非功能需求变化 | 架构 + 详设 | 架构非功能设计章节变化 |
| 数据模型变更 | 详设(后端) + 单测 + 测试用例 | 详设DDL/ER图变化 |
| 权限模型变更 | PRD + 详设(前端+后端) + 测试用例 | PRD角色权限矩阵变化 |

> **AI生成规则**：
> - 影响分析必须基于实际已有产物（无该环节的产物则标注"无已有产物，需新建"）
> - 影响链中每个环节须标注具体受影响的章节/页面/接口
> - 不确定是否受影响时，标注"⚠️ 待确认"并询问用户

#### 输出影响分析表

| 环节 | 对应 Skill | 是否受影响 | 影响内容 | 已有产物 |
|------|-----------|:---------:|---------|---------|
| PRD | lyspec-prd | ✅ / ❌ | {具体受影响的章节/用例} | {文件路径或"无"} |
| UED | lyspec-ued | ✅ / ❌ | {具体受影响的页面} | {文件路径或"无"} |
| 架构设计 | lyspec-arch | ✅ / ❌ | {具体受影响的章节} | {文件路径或"无"} |
| 详细设计-后端 | lyspec-backend | ✅ / ❌ | {具体受影响的接口/类/表} | {文件路径或"无"} |
| 详细设计-前端 | lyspec-frontend | ✅ / ❌ | {具体受影响的页面/组件} | {文件路径或"无"} |
| 代码编写 | — | ✅ / ❌ | {需修改的代码模块} | — |
| 单元测试 | lyspec-unit-test | ✅ / ❌ | {需新增/修改的测试} | — |
| 测试用例 | lyspec-test-cases | ✅ / ❌ | {需新增/修改的用例类别} | {文件路径或"无"} |

### 第三步：生成变更产物

#### 3.1 创建变更目录

```bash
mkdir -p output/changes/<change-name>
```

#### 3.2 生成 change-proposal.md

```markdown
# 变更提案：{变更名称}

## 基本信息

| 字段 | 内容 |
|------|------|
| 变更名称 | {changeName} |
| 项目名称 | {projectName} |
| 提案日期 | {YYYY-MM-DD} |
| 变更原因 | {客户反馈/业务调整/技术限制/...} |
| 变更优先级 | 高/中/低 |

## 变更描述

{对变更内容的完整描述，包括改什么、为什么改}

## 变更范围

### 涉及的功能模块
- {模块1}：{变更内容}
- {模块2}：{变更内容}

### 不涉及的功能模块
- {模块X}：不受影响，原因：{说明}

## 影响分析

{影响分析表，同第二步输出}

## 风险评估

| 风险 | 等级 | 应对措施 |
|------|------|---------|
| {风险1} | 高/中/低 | {措施} |

## 关联变更

| 关联类型 | 变更名称 | 说明 |
|---------|---------|------|
| 前置变更 | {如有} | {说明} |
| 并行变更 | {如有} | {说明} |
```

#### 3.3 生成 change-tasks.md

```markdown
# 变更任务清单：{变更名称}

> 基于影响分析自动生成，按执行顺序排列。

## 执行顺序

{仅列出受影响的环节，未受影响的环节不出现}

- [ ] **① PRD增量更新**（lyspec-prd 增量更新模式）
  - 更新内容：{具体章节和内容}
  - 已有产物：{文件路径}

- [ ] **② UED更新**（lyspec-ued）
  - 更新内容：{具体页面和组件}
  - 已有产物：{文件路径}

- [ ] **③ 架构设计更新**（lyspec-arch）
  - 更新内容：{具体章节}
  - 已有产物：{文件路径}

- [ ] **④ 详细设计更新**（lyspec-detail）
  - 后端更新：{接口/类/表}
  - 前端更新：{页面/组件}
  - 已有产物：{文件路径}

- [ ] **⑤ 代码修改**
  - 修改内容：{代码模块}

- [ ] **⑥ 单元测试更新**（lyspec-unit-test，强制）
  - 更新内容：{新增/修改的测试}

- [ ] **⑦ 测试用例更新**（lyspec-test-cases）
  - 更新内容：{新增/修改的用例类别}
  - 已有产物：{文件路径}
```

> **生成规则**：
> - 任务顺序遵循 LYSpec 的上下游依赖关系（PRD → UED → 架构 → 详设 → 代码 → 单测 → 测试用例）
> - 仅列出影响分析中标记为"✅ 受影响"的环节
> - 每个任务须标注对应的 skill 和已有产物路径
> - 单元测试（⑥）在涉及代码变更时始终包含（强制执行）

### 第四步：用户确认

输出变更提案摘要，暂停等待用户确认：

```
## 变更提案摘要

**变更名称**：{changeName}
**变更原因**：{原因}
**影响范围**：{受影响环节数} / 8 个环节

### 受影响环节
1. {环节1}：{影响内容摘要}
2. {环节2}：{影响内容摘要}
...

### 产物已生成
- output/changes/{changeName}/change-proposal.md
- output/changes/{changeName}/change-tasks.md

请确认：
1. 变更范围是否准确？
2. 影响分析是否有遗漏？
3. 确认后执行 lyspec-change-apply 开始变更实施
```

## 特殊场景处理

### 同名变更已存在

若 `output/changes/<change-name>/` 已存在：
- 询问用户："发现同名变更已存在，是要继续该变更（更新提案），还是新建一个？"
- 继续 → 读取已有提案，在其基础上修改
- 新建 → 使用 `<change-name>-v2` 等命名

### 多个变更并行

支持同时存在多个活跃变更：
- `output/changes/add-export-feature/`
- `output/changes/modify-approval-flow/`

每个变更独立管理，互不影响。

### 变更依赖已归档的变更

若本次变更依赖某个已归档的变更：
- 在 change-proposal.md 的"关联变更"中标注
- 提醒用户确认前置变更已合并到主产物中

## 附录

### A. 用户交互检查点

- [ ] 变更背景已充分了解（改什么、为什么改）
- [ ] 已有产物已扫描和阅读
- [ ] 变更范围已与用户澄清确认
- [ ] 影响分析已完成并确认
- [ ] change-proposal.md 已生成
- [ ] change-tasks.md 已生成
- [ ] 用户确认提案，准备进入执行

### B. 质量检查点

**影响分析完整性**：
- [ ] 每个受影响环节标注了具体的受影响内容（非泛化描述）
- [ ] 影响链中没有遗漏的下游环节（PRD变了 → 测试用例必须受影响）
- [ ] 不确定的影响项已标注"⚠️ 待确认"
- [ ] 已有产物路径正确且文件存在

**任务清单完整性**：
- [ ] 任务顺序符合上下游依赖（不出现先改详设后改PRD的情况）
- [ ] 每个任务标注了对应的 skill
- [ ] 涉及代码变更时，单元测试任务已包含
- [ ] 每个任务的"更新内容"足够具体，lyspec-change-apply 可据此执行
