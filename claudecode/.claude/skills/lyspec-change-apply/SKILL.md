# 需求变更执行 Skill（AI友好版）

> 本文档处于 LYSpec 变更控制流程的**中间环节**："变更提案→**变更执行**→变更归档"。
> 变更执行按 `change-tasks.md` 中的任务清单，选择性调用现有 lyspec-* skill 完成变更实施。
> 不是重走全链路，而是只触发受影响的环节。

---

## 定位与边界

| 维度 | lyspec-change-apply（本 skill） | lyspec-orchestrator A路径 |
|------|----------------------------------|---------------------------|
| 执行范围 | 仅受影响的环节 | 全链路顺序执行 |
| 调用方式 | 选择性调用现有 skill | 顺序调用所有 skill |
| 产物模式 | 增量更新已有产物 | 生成全新产物 |
| 进度跟踪 | change-tasks.md 勾选状态 | orchestrator 逐步提示 |

---

## AI执行流程

### 第一步：选择变更

**1.1 确定目标变更**

- 若用户指定了变更名称 → 直接使用
- 若用户在对话中提及某变更 → 从上下文推断
- 若 `output/changes/` 下仅存在一个活跃变更 → 自动选中
- 若存在多个活跃变更 → 列出可用变更，询问用户选择

```
当前活跃变更：
1. add-export-feature（提案日期：2026-03-10）
2. modify-approval-flow（提案日期：2026-03-11）

请选择要执行的变更（输入编号或名称）：
```

始终说明："当前使用变更：{changeName}"。

**1.2 读取变更产物**

读取以下文件：
- `output/changes/<changeName>/change-proposal.md` — 理解变更背景和影响分析
- `output/changes/<changeName>/change-tasks.md` — 获取任务清单

若文件不存在，提示："未找到变更提案，请先执行 lyspec-change-propose 创建变更提案。"

---

### 第二步：展示当前进度

读取 `change-tasks.md` 并统计任务状态：

```
## 变更执行：{changeName}

**进度**：{已完成数}/{总任务数} 任务已完成

### 待执行任务
- [ ] ① PRD增量更新 — 更新第四章用例 UC-005
- [ ] ④ 详细设计更新 — 新增审批会签接口
- [ ] ⑥ 单元测试更新 — 补充会签逻辑测试
- [ ] ⑦ 测试用例更新 — 新增会签流程测试场景

### 已完成任务
（无）
```

---

### 第三步：逐任务执行

按 `change-tasks.md` 中的顺序，逐个执行任务。

#### 执行规则

**对每个待执行任务**：

1. **声明当前任务**：
   ```
   正在处理任务 {N}/{M}：{任务描述}
   ```

2. **调用对应 skill**：

   | 任务环节 | 调用的 Skill | 调用模式 | 说明 |
   |---------|-------------|---------|------|
   | PRD增量更新 | lyspec-prd | 增量更新模式 | 基于已有PRD调整，生成新版本 |
   | UED更新 | lyspec-ued | 针对受影响页面重新设计 | 仅更新受影响的页面 |
   | 架构设计更新 | lyspec-arch | 针对受影响章节更新 | 通常仅更新非功能设计或模块划分 |
   | 详细设计更新 | lyspec-detail | 针对受影响模块更新 | 按需调用 backend/frontend 子skill |
   | 代码修改 | — | 直接编码 | 基于更新后的详细设计修改代码 |
   | 单元测试更新 | lyspec-unit-test | 测试生成/修复模式 | 强制执行，不可跳过 |
   | 测试用例更新 | lyspec-test-cases | 增量生成 | 基于变更范围补充/修改测试用例 |
   | 代码评审 | lyspec-code-review | 增量评审模式 | 仅评审变更涉及的代码，强制执行 |

3. **增量更新原则**：
   - 调用 skill 时，明确告知 skill 当前是**变更更新模式**，非全新生成
   - 提供已有产物路径，让 skill 在已有产物基础上修改
   - 仅更新受影响的部分，不触动未受影响的内容

4. **完成后更新状态**：
   - 在 `change-tasks.md` 中将该任务标记为完成：`- [ ]` → `- [x]`
   - 在任务项下追加完成摘要：
     ```markdown
     - [x] **① PRD增量更新**（lyspec-prd 增量更新模式）
       - 更新内容：第四章新增 UC-012 会签用例
       - 已有产物：output/项目A_PRD_2026-03-08.md
       - ✅ 完成于 2026-03-11，更新文件：output/项目A_PRD_2026-03-11.md
     ```

5. **继续下一个任务**

#### 暂停机制

以下情况**暂停执行**，不继续下一个任务：

**场景1：任务不清晰**
```
## 执行暂停

**变更**：{changeName}
**进度**：{N}/{M} 任务已完成
**当前任务**：{任务描述}

### 问题
任务描述不够清晰，无法确定具体的修改范围。

**需要澄清**：{具体问题}

请补充信息后继续。
```

**场景2：发现设计冲突**
```
## 执行暂停

**变更**：{changeName}
**进度**：{N}/{M} 任务已完成
**当前任务**：{任务描述}

### 发现设计冲突
在执行当前任务时发现：{冲突描述}

**建议**：
1. 回到变更提案，修订影响分析
2. 调整当前任务的执行方式：{建议}
3. 跳过当前任务，继续后续任务

您希望怎么做？
```

**场景3：执行出错**
```
## 执行暂停

**变更**：{changeName}
**进度**：{N}/{M} 任务已完成
**当前任务**：{任务描述}

### 遇到错误
{错误描述}

**可选操作**：
1. {修复方案1}
2. {修复方案2}
3. 跳过当前任务，继续后续

您希望怎么做？
```

**场景4：用户主动打断**
- 记录当前进度到 `change-tasks.md`
- 输出当前状态摘要
- 提示用户："下次可通过 lyspec-change-apply 继续执行变更 '{changeName}'。"

## 与现有 Skill 的调用约定

### 调用时传递的上下文

调用每个现有 skill 时，须提供以下上下文信息：

```
【变更上下文】
- 变更名称：{changeName}
- 变更模式：增量更新（非全新生成）
- 已有产物路径：{filePath}
- 本次更新范围：{具体受影响的章节/页面/接口}
- 变更提案参考：output/changes/{changeName}/change-proposal.md
```

### 各 Skill 的增量更新行为

| Skill | 增量更新行为 |
|-------|------------|
| lyspec-prd | 使用"增量更新模式"，基于已有PRD修改受影响章节，文件名加新日期 |
| lyspec-ued | 仅重新设计受影响的页面，不触动其他页面的 HTML Demo |
| lyspec-arch | 仅更新受影响的章节（如模块划分、非功能设计），保留其余章节 |
| lyspec-detail | 仅更新受影响的接口/类/表设计，保留其余设计 |
| lyspec-unit-test | 为变更涉及的代码生成/更新单元测试 |
| lyspec-test-cases | 为变更涉及的功能/接口补充/修改测试用例 |
| lyspec-code-review | 使用"增量评审模式"，仅评审变更涉及的代码文件，强制执行 |

## 附录

### A. 用户交互检查点

- [ ] 目标变更已确认
- [ ] 变更提案和任务清单已读取
- [ ] 当前进度已展示
- [ ] 每个任务执行前已声明
- [ ] 每个任务完成后已更新 change-tasks.md
- [ ] 执行完成或暂停时已输出状态摘要

### B. 质量检查点

**执行完整性**：
- [ ] 所有标记为受影响的环节均已执行
- [ ] 单元测试在涉及代码变更时已执行（不可跳过）
- [ ] 每个任务的完成摘要已记录到 change-tasks.md

**产物一致性**：
- [ ] 更新后的PRD与详细设计保持一致（功能点对应、术语一致）
- [ ] 更新后的详细设计与测试用例保持一致（接口/错误码覆盖）
- [ ] 更新后的前端设计与后端接口保持一致（URL/参数/响应体）

**变更隔离性**：
- [ ] 仅修改了受影响的部分，未触动不相关的内容
- [ ] 更新后的产物文件可独立使用（不出现半完成状态）
