# 功能测试用例生成模板（AI友好版）

> 基于企业测试用例规范优化，面向AI辅助生成场景。
> 本文档处于软件生命周期的"详细设计→代码编写→**测试用例**"阶段，承接上游设计产物。
> 支持多维度输入（PRD + 详细设计文档 + 架构设计文档），根据输入丰富度自动扩展测试维度。

---

## 全局规则

### 输出格式规范

| 内容类型 | 格式要求 |
|----------|----------|
| 用例集合 | JSON 数组 + Excel 双格式输出 |
| 进度报告 | 表格（维度、数量、覆盖率） |
| 需求覆盖矩阵 | 表格（需求ID × 用例ID 交叉） |
| 中间结果 | JSON（仅用户指定 `--intermediate` 时输出） |

### 信息提取三级策略

| 级别 | 条件 | 处理方式 |
|------|------|----------|
| ① 直接提取 | 输入文档中有明确描述（PRD功能点、详细设计接口定义等） | 直接生成用例，来源标注文档名称和章节 |
| ② AI推导 | 输入文档中无直接描述，但可从上下文推导（如边界值、异常路径） | AI推导生成，标注"AI推导"，**用斜体标识** |
| ③ 待补充 | 完全无依据，但测试经验表明应覆盖 | 标注"建议补充"，在用户确认环节集中询问 |

### 术语一致性约束

- 用例中的业务术语必须与PRD和详细设计文档中的术语一致
- 页面名称、接口名称、字段名称必须与上游文档完全匹配，不可自行简化或改写

### 用例编号规则

| 用例类型 | ID格式 | 示例 |
|---------|--------|------|
| 页面功能用例 | `TC-{模块缩写}-P{页面序号}-{三位序号}` | `TC-UM-P01-001` |
| 业务流程用例 | `TC-{模块缩写}-F{流程序号}-{三位序号}` | `TC-UM-F01-001` |
| 接口测试用例 | `TC-{模块缩写}-A{接口序号}-{三位序号}` | `TC-UM-A01-001` |
| 全局约束用例 | `TC-{模块缩写}-G-{三位序号}` | `TC-UM-G-001` |

> 模块缩写由用户指定或从PRD模块名称自动生成（取首字母大写）。

## 前置条件：上游文档输入

> **测试用例的质量取决于输入文档的丰富度。**

### 输入文档与测试维度对应关系

| 输入文档 | 必需/可选 | 可生成的测试维度 |
|---------|-----------|-----------------|
| PRD（产品需求文档） | **必需** | 页面功能测试、业务流程测试、权限测试、UI交互测试 |
| 详细设计文档（lyspec-detail产物） | **推荐** | 接口参数校验、错误码验证、幂等性测试、数据库约束、缓存一致性 |
| 架构设计文档（lyspec-arch产物） | 可选 | 非功能测试场景（性能/安全/兼容性）、集成测试场景 |
| UI/UE设计稿 | 可选 | UI一致性测试、响应式测试 |

### AI执行流程

1. **确认输入文档**：

> 请提供以下文档（至少提供PRD）：
>
> | 文档 | 路径 | 状态 |
> |------|------|------|
> | PRD（必需） | ? | — |
> | 详细设计文档（推荐） | ? | — |
> | 架构设计文档（可选） | ? | — |
> | UI/UE设计稿（可选） | ? | — |
>
> 提供的文档越丰富，生成的测试用例覆盖度越高。

2. **读取文档并提取关键信息**：

   **从PRD提取**：
   - 页面清单（含弹窗、子页面）
   - 业务流程（主流程、子流程、异常流程）
   - 角色权限矩阵
   - 业务规则和约束条件
   - 输入校验规则（字段长度、格式、枚举值）

   **从详细设计文档提取**（如提供）：
   - 接口清单及参数定义（第四章 4.3~4.4节）
   - 错误码定义（第四章 4.2节）
   - 幂等性设计（第四章 4.3节幂等性列）
   - 数据库字段约束（第七章 7.2节）
   - 缓存策略（第七章 7.4节）
   - 外部依赖及降级方案（第五章 5.1节）

   **从架构设计文档提取**（如提供）：
   - 性能要求（第10.2节）
   - 安全要求（第16.3节）
   - 兼容性要求（第16.10节）

3. **确定测试维度**：根据实际提供的文档，自动确定可生成的测试维度，并告知用户：
   > "根据您提供的文档，本次可生成以下测试维度：{列表}。以下维度因缺少输入文档暂无法生成：{列表}。"

## 生成模式选择

| 模式 | 说明 | 适用场景 |
|------|------|----------|
| **逐步确认模式**（默认） | 分4个Stage执行，每Stage暂停等待用户确认后再继续 | 核心模块、首次使用、PRD结构不标准 |
| **快速生成模式** | 连续执行所有Stage，仅在最终输出后确认 | 简单模块、PRD结构清晰规范 |

> 若用户未明确选择，默认使用逐步确认模式。

## 四阶段流水线

严格按以下四个阶段顺序执行。

---

### Stage 1：文档解析（拆解输入文档）

> **目标**：将输入文档拆解为结构化数据，作为后续用例生成的基础。

#### 1-1 页面信息提取

从PRD中提取所有页面（含弹窗、子页面）：

| 字段 | 说明 |
|------|------|
| pageId | 页面编号，格式 `PAGE-{三位序号}` |
| pageName | 页面名称（与PRD完全一致） |
| pageType | 列表页 / 表单页 / 详情页 / 弹窗 / 抽屉 / 向导页 |
| accessPermissions | 可访问角色列表 |
| entryPaths | 入口路径（从哪些页面/操作可到达） |
| pageUrl | 路由路径，弹窗设为 null |
| components | 页面包含的组件类型（输入框/下拉框/表格/按钮等） |

**提取规则**：
- 扫描PRD中所有页面描述、原型图说明、页面基础信息表格
- 弹窗、抽屉、子页面不可遗漏，每个独立交互区域算一个页面
- `components` 字段用于Stage 2判断适用的测试类别
- `entryPaths` 始终为数组，无入口时为空数组

> **AI生成规则**：
> - 页面名称必须与PRD中的名称完全一致，不可简化
> - 页面类型根据PRD描述推断，不确定时标注并询问用户
>
> **常见错误**：
> - ❌ 遗漏弹窗和抽屉 → 弹窗/抽屉是独立测试对象，必须提取
> - ❌ 将Tab页签当作独立页面 → Tab页签属于同一页面的不同状态
> - ❌ 页面名称与PRD不一致 → 影响需求覆盖矩阵的追溯

#### 1-2 业务流程提取

提取所有业务流程（主流程、子流程、异常流程）：

| 字段 | 说明 |
|------|------|
| processId | 流程编号，格式 `FLOW-{三位序号}` |
| processName | 流程名称 |
| triggerCondition | 触发条件 |
| priority | 优先级：核心/重要/一般 |
| normalFlow | 正常流程步骤数组（每步含：步骤ID、操作描述、预期结果、关联页面） |
| exceptionFlows | 异常流程数组（每条含：异常ID、触发条件、处理方式） |

**提取规则**：
- 从PRD的"业务流程"、"操作流程"章节提取
- 正常流程按步骤顺序，每步关联到具体页面
- 异常流程从"异常流程表"或异常场景描述中提取
- 子流程/分支流程独立为单独的流程对象
- 为每个流程标注优先级：涉及核心业务链路的为"核心"，辅助功能为"一般"

> **常见错误**：
> - ❌ 只提取正常流程，忽略异常流程 → 异常流程是测试重点
> - ❌ 流程步骤粒度太粗（"完成操作"） → 每步必须是可执行的具体操作

#### 1-3 接口信息提取（需详细设计文档）

> **跳过条件**：未提供详细设计文档时跳过

从详细设计文档提取接口定义：

| 字段 | 说明 |
|------|------|
| apiId | 接口编号，格式 `API-{三位序号}` |
| apiName | 接口名称 |
| method | HTTP方法 |
| url | 接口URL |
| params | 参数列表（含：名称、类型、必填、校验规则、枚举值） |
| errorCodes | 关联的错误码列表 |
| idempotency | 幂等性保障方式 |

**提取规则**：
- 从详细设计文档第四章 4.3~4.4节逐条提取
- 参数校验规则必须完整提取（长度、范围、格式、枚举值）
- 错误码从第四章 4.2节关联

#### 1-4 全局约束提取

从所有输入文档中提取跨页面/跨流程的全局约束：

| 约束类型 | 提取来源 |
|---------|---------|
| 权限矩阵 | PRD角色权限章节 |
| 错误处理规则 | PRD通用规则 + 详细设计4.2节错误码 |
| 接口规范 | 详细设计4.1节接口协议 |
| 缓存策略 | 详细设计7.4节缓存设计 |
| 性能要求 | 架构设计文档10.2节 |
| 安全要求 | 架构设计文档16.3节 |
| 兼容性要求 | 架构设计文档16.10节 |

**Stage 1完成后**，向用户报告并**暂停等待确认**：

> **Stage 1 解析完成**：
>
> | 维度 | 数量 | 状态 |
> |------|------|------|
> | 页面 | X 个 | 已提取 |
> | 业务流程 | Y 个（核心 M 个） | 已提取 |
> | 接口定义 | Z 个 | 已提取 / 未提供详细设计 |
> | 全局约束 | W 类 | 已提取 |
>
> 请确认：
> 1. 页面和流程是否有遗漏？
> 2. 是否需要补充或调整？
> 3. 确认后进入 Stage 2（用例框架生成）

---

### Stage 2：测试用例框架生成（用例名称 + 优先级）

> **目标**：基于Stage 1的结构化数据，生成每个测试维度的用例名称和优先级。

#### 2-1 页面功能测试框架

为每个页面生成测试用例名称，按以下类别组织：

| 类别 | 英文key | 适用条件 | 最少用例数指引 |
|------|---------|----------|---------------|
| 布局与样式 | `layoutStyle` | 所有页面 | 2~3条 |
| 元素存在性与状态 | `elementExistence` | 所有页面 | 每个关键元素1条 |
| 文字内容 | `textContent` | 所有页面 | 2~3条 |
| 图片与多媒体 | `imageMultimedia` | 有图片/音视频时 | 每种媒体类型1条 |
| 输入框测试 | `inputField` | 有输入框时 | 每个输入框至少3条（正常/边界/异常） |
| 下拉框/选择器 | `dropdownSelector` | 有下拉框时 | 每个下拉框至少2条 |
| 按钮与操作 | `buttonOperation` | 有按钮时 | 每个按钮至少2条（正常/禁用态） |
| 表单提交与重置 | `formSubmitReset` | 有表单时 | 至少4条（成功提交/校验失败/重置/重复提交） |
| 数据展示 | `dataDisplay` | 有数据展示时 | 至少3条（正常/空数据/大量数据） |
| 数据操作 | `dataOperation` | 有增删改查时 | 每个操作至少2条（正常/异常） |
| 数据边界 | `dataBoundary` | 有列表/分页时 | 至少3条（首页/末页/超出范围） |
| 页面状态 | `pageState` | 有状态切换时 | 每种状态转换1条 |
| 用户登录状态 | `loginState` | 有权限要求时 | 至少2条（已登录/未登录） |
| 多步骤流程 | `multiStepProcess` | 属于多步骤流程时 | 至少3条（完成/中断/回退） |
| **业务规则验证** | `businessRule` | 有业务约束时 | 每条规则至少2条（满足/违反） |
| **数据联动** | `dataLinkage` | 有联动关系时 | 每组联动关系至少2条 |

**命名规范**：`{页面名称}-{类别描述}-{具体场景}`

**用例优先级判定规则**：

| 优先级 | 判定条件 | 说明 |
|--------|---------|------|
| P0（冒烟） | 核心业务主路径上的验证点 | 部署后必须验证 |
| P1（高） | 重要功能的正向验证 + 核心异常路径 | 每轮回归必须执行 |
| P2（中） | 非核心功能验证 + 边界值测试 | 全量回归时执行 |
| P3（低） | UI细节、极端边界、兼容性 | 有时间时执行 |

> **AI生成规则**：
> - 每个功能点至少对应一个用例名称
> - 页面无对应组件的类别跳过（不输出空数组）
> - 用例名称必须具体到场景，不可使用泛化描述
> - 每个用例必须标注优先级（P0/P1/P2/P3）
> - P0用例数量控制在总数的10~15%
>
> **常见错误**：
> - ❌ 给每个输入框生成完全相同的"输入为空"用例 → 应根据字段业务含义生成差异化用例
> - ❌ 用例名称过于泛化（"验证功能正常"） → 必须具体到操作和预期
> - ❌ 只生成正向用例 → 每个功能点至少有1个正向和1个反向用例
> - ❌ 所有用例都标P1 → 严格按判定规则区分优先级

#### 2-2 业务流程测试框架

为每个流程生成三类测试用例名称：

| 类别 | 英文key | 说明 | 最少用例数指引 |
|------|---------|------|---------------|
| 正常流程 | `normalFlow` | 主干流程的每步验证 | 流程步骤数 × 1 |
| 异常流程 | `exceptionFlow` | 每个异常分支的验证 | 异常分支数 × 1 |
| 流程跳转 | `flowJump` | 流程间跳转、中断恢复、回退 | 至少2条 |

> **AI生成规则**：
> - 核心流程（priority=核心）的用例优先级为 P0~P1
> - 异常流程的用例不可少于正常流程
> - 流程跳转必须覆盖：正常跳转、中断后恢复、浏览器后退、强制刷新
>
> **常见错误**：
> - ❌ 异常流程用例数量远少于正常流程 → 异常场景是测试重点
> - ❌ 遗漏流程中断场景（如网络断开、会话超时） → 必须覆盖

#### 2-3 接口测试框架（需详细设计文档）

> **跳过条件**：未提供详细设计文档时跳过

为每个接口生成测试用例名称：

| 类别 | 英文key | 说明 | 最少用例数指引 |
|------|---------|------|---------------|
| 参数校验 | `paramValidation` | 必填/长度/格式/枚举/类型 | 每个参数至少2条 |
| 正常响应 | `normalResponse` | 成功场景的响应验证 | 至少1条 |
| 错误码验证 | `errorCodeVerify` | 每个错误码的触发场景 | 每个错误码1条 |
| 幂等性验证 | `idempotencyVerify` | 重复请求不产生重复数据 | 非GET接口各1条 |
| 边界值 | `boundaryValue` | 参数的边界值测试 | 有范围约束的参数各2条 |
| 权限验证 | `authVerify` | 不同角色的访问控制 | 每个角色1条 |

> **AI生成规则**：
> - 参数校验用例从详细设计4.4节的"校验规则"列逐条推导
> - 错误码用例从详细设计4.2节的错误码表逐条推导
> - 幂等性用例从详细设计4.3节的"幂等性"列推导
>
> **常见错误**：
> - ❌ 只测试必填校验，忽略长度/格式/枚举校验 → 每种校验规则都需要用例
> - ❌ 只测试200响应，不测试4xx/5xx响应 → 必须覆盖所有错误码

#### 2-4 全局约束测试框架

按输入文档丰富度动态生成：

| 类别 | 英文key | 需要的输入 | 说明 |
|------|---------|-----------|------|
| 权限测试 | `permissionTests` | PRD权限矩阵 | 角色 × 功能的交叉验证 |
| 错误处理测试 | `errorHandlingTests` | PRD / 详细设计 | 通用错误处理机制验证 |
| 缓存测试 | `cacheTests` | 详细设计7.4节 | 缓存命中/失效/一致性 |
| 接口通用测试 | `interfaceCommonTests` | 详细设计4.1节 | 认证、限流、统一响应格式 |
| 安全测试场景 | `securityTests` | 架构设计16.3节 | XSS/SQL注入/CSRF/越权 |
| 性能测试场景 | `performanceTests` | 架构设计10.2节 | 响应时间/并发/TPS指标验证 |
| 兼容性测试 | `compatibilityTests` | 架构设计16.10节 | 浏览器/设备兼容 |
| 数据库约束测试 | `dbConstraintTests` | 详细设计7.2节 | 唯一约束/NOT NULL/字段长度 |

> **AI生成规则**：
> - 仅在对应输入文档提供时才生成该类别
> - 安全测试至少覆盖OWASP Top 5：SQL注入、XSS、CSRF、越权访问、敏感数据泄露
> - 性能测试场景只生成场景描述和验收指标，不生成执行脚本

#### 2-5 需求覆盖矩阵

Stage 2完成后，输出需求覆盖矩阵，验证PRD中每个功能点至少有1个测试用例覆盖：

| PRD功能点 | PRD章节 | 覆盖用例ID列表 | 覆盖数 | 状态 |
|-----------|---------|---------------|--------|------|
| {功能点1} | {章节} | TC-xx-001, TC-xx-005 | 2 | 已覆盖 |
| {功能点2} | {章节} | — | 0 | **未覆盖** |

> **AI生成规则**：
> - 未覆盖的功能点必须醒目标注，并补充对应用例
> - 覆盖矩阵输出后暂停，等待用户确认覆盖率是否满足要求

**Stage 2完成后**，向用户报告并**暂停等待确认**：

> **Stage 2 框架生成完成**：
>
> | 维度 | 用例数 | P0 | P1 | P2 | P3 |
> |------|--------|----|----|----|----|

### Stage 3：测试用例详细填充

> **目标**：为Stage 2的每个用例名称填充完整的测试步骤和预期结果。

#### 用例字段规范

| 字段 | 必填 | 说明 |
|------|------|------|
| testCaseId | 是 | 用例编号（见全局规则中的编号规则） |
| testCaseName | 是 | 用例名称（来自Stage 2） |
| priority | 是 | P0 / P1 / P2 / P3 |
| level | 是 | 冒烟 / 基本 / 详细 |
| relatedRequirement | 是 | 关联的PRD功能点或需求ID |
| precondition | 否 | 前置条件（可为空） |
| testSteps | 是 | 测试步骤数组，每步为具体可执行的操作 |
| expectedResult | 是 | 预期结果，必须为可验证的具体描述 |
| testData | 否 | 测试数据（可为空） |
| source | 是 | 用例来源：PRD / 详细设计 / AI推导 |

**level 与 priority 的对应关系**：

| priority | level | 含义 |
|----------|-------|------|
| P0 | 冒烟 | 核心路径验证，部署后立即执行 |
| P1 | 基本 | 重要功能验证，每轮回归执行 |
| P2 | 详细 | 完整功能覆盖，全量回归执行 |
| P3 | 详细 | 极端场景覆盖，有时间时执行 |

#### 填充原则

1. **正反用例**：功能验证类同时覆盖正常和异常场景
2. **边界值**：有长度/数值限制的字段测试边界值（最小值、最大值、最小值-1、最大值+1）
3. **枚举全覆盖**：有枚举值的字段覆盖所有选项
4. **异常场景**：参考PRD异常流程表 + 详细设计错误码表补充
5. **数据驱动**：数据展示类覆盖空数据、单条数据、正常数据、大量数据（超分页）
6. **步骤可执行**：每步描述具体操作和操作对象，无歧义

> **AI生成规则**：
> - 测试步骤必须是具体的可执行操作（如"在用户名输入框中输入'admin'"），不可使用泛化描述（如"输入正确的用户名"）
> - 预期结果必须是可验证的具体描述（如"页面跳转到/dashboard，顶部显示'欢迎admin'"），不可使用"成功"、"正确"等模糊描述
> - 测试数据应使用示例值（如"admin"、"123456"），不可使用占位符（如"xxx"）
> - 每个用例的步骤数控制在3~8步，超过8步应拆分为多个用例
>
> **常见错误**：
> - ❌ 测试步骤写"验证功能正常" → 必须写具体操作
> - ❌ 预期结果写"成功" → 必须写具体的可验证现象
> - ❌ 测试数据写"正确的手机号" → 必须写具体值如"13800138000"
> - ❌ 单个用例步骤超过10步 → 应拆分为多个用例
> - ❌ 多个用例的步骤和预期结果完全相同只是数据不同 → 合并为数据驱动用例，在testData中列出多组数据

**Stage 3完成后**，向用户报告并**暂停等待确认**：

> **Stage 3 用例填充完成**：
>
> | 维度 | 用例总数 | P0(冒烟) | P1(基本) | P2(详细) | P3(详细) |
> |------|---------|---------|---------|---------|---------|
> | 页面功能 | X | a | b | c | d |
> | 业务流程 | Y | a | b | c | d |
> | 接口测试 | Z | a | b | c | d |
> | 全局约束 | W | a | b | c | d |
> | **合计** | **N** | — | — | — | — |
>
> 请确认：
> 1. 用例质量是否满足要求？
> 2. 是否需要补充或调整？

### B. 质量检查点

**需求覆盖性**：
- [ ] PRD中每个功能点至少有1个测试用例覆盖

- [ ] 每个用例的测试步骤是具体可执行的操作（非泛化描述）
- [ ] 每个用例的预期结果是可验证的具体描述（非"成功"）

- [ ] 页面名称与PRD完全一致
