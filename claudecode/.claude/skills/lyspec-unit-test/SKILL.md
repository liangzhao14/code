# 单元测试 Skill（AI友好版）

> 本文档处于软件生命周期的"代码编写→**单元测试**→功能测试(lyspec-test-cases)"阶段。
> 单元测试关注**代码级**的方法/函数/组件行为验证，与功能测试（lyspec-test-cases）互补。
> 测试框架规范通过 references/ 按技术栈按需加载，避免一次性消耗过多 context。

---

## 定位与边界

### 与 lyspec-test-cases 的关系

| 维度 | lyspec-unit-test（本 skill） | lyspec-test-cases |
|------|-------------------------------|-------------------|
| 测试层次 | 代码级（函数/方法/组件） | 功能级（用户场景/接口/E2E） |
| 输入来源 | 源代码 + 详细设计（类图、异常体系） | PRD + 详细设计（接口清单、错误码） |
| 输出产物 | 测试代码文件（`*.test.ts` / `*Test.java`） | 测试用例文档（Excel/Markdown） |
| 执行方式 | CI 自动执行 | 手工执行 + 自动化脚本 |
| 覆盖目标 | 代码行/分支覆盖率 ≥ 80% | 功能点/业务场景覆盖率 |

两者互补，不重叠。

## 全局规则

### 覆盖率目标

| 层次 | 目标 | 说明 |
|------|------|------|
| 行覆盖率 | ≥ 80% | 核心业务模块 |
| 分支覆盖率 | ≥ 70% | 含条件判断的业务逻辑 |
| 方法覆盖率 | 100% | 所有 public 方法至少一个测试 |

> 工具类/配置类等低风险代码可适当降低要求，但须向用户说明。

### 测试命名规范

| 技术栈 | 命名格式 | 示例 |
|--------|---------|------|
| Java | `should_期望行为_when_前置条件` | `should_throwException_when_nameDuplicate` |
| 前端 | 中文描述 `describe('模块') → it('应正确处理...')` | `it('应在输入为空时返回默认值')` |

### 核心原则

| 原则 | 说明 |
|------|------|
| **不可跳过** | 单元测试为强制环节，必须生成测试代码并执行通过后才能进入下游环节 |
| 不启动运行时容器 | Java 禁止 `@SpringBootTest`；前端不依赖浏览器环境 |
| 隔离外部依赖 | 所有 DB、HTTP、MQ、文件系统均 Mock |
| 每个测试只验证一个行为 | 遵循 AAA 模式（Arrange-Act-Assert） |
| 测试之间无状态共享 | 每个 `@Test` / `it()` 独立，不依赖执行顺序 |
| 只修复测试代码 | 测试失败时优先修复测试，不为通过测试而修改业务代码 |

> **强制约束**：单元测试不可跳过。代码编写完成后必须执行本skill，生成单元测试并确保全部通过，方可进入功能测试（lyspec-test-cases）或代码评审环节。

### 输出格式规范

| 内容类型 | 格式要求 |
|----------|----------|
| 测试策略表 | 必须使用**表格**（方法名、正常路径、边界条件、异常路径） |
| 测试代码 | 必须使用**代码块**，标注语言类型 |
| 覆盖率报告 | 必须使用**表格**（方法 × 场景矩阵） |
| 失败分析 | 必须使用**分点列表**（错误类型→原因→修复方式） |

## 前置条件：上游输入

### 上游依赖

| 输入来源 | 必需/可选 | 提取内容 |
|---------|-----------|----------|
| 被测源代码 | **必需** | public 方法、依赖注入、逻辑分支、异常抛出 |
| 后端详细设计（lyspec-backend） | 推荐 | 分层架构（4.2节）→逐层测试策略；异常体系（5.2节）→异常路径覆盖；事务设计（5.3节）→事务边界测试 |
| 前端详细设计（lyspec-frontend） | 推荐 | 组件树（3.1节）→组件测试清单；状态管理（5.1节）→Store测试；API层（4.4节）→Mock策略 |
| 全栈详细设计（lyspec-detail） | 可选 | 接口设计 + 错误码 → 参数校验测试、异常返回测试 |

### AI执行流程

1. **确认被测目标**：
   - "请指定需要生成单测的文件/模块路径"
   - 若用户未指定，从代码仓库识别尚未覆盖的 Service/组件

2. **确认技术栈**（若未明确）：
   - "当前项目使用什么测试框架？"
   - 根据 `pom.xml` / `package.json` 自动识别

3. **加载对应 reference**：根据技术栈加载测试规范

4. **提取测试上下文**（若有详细设计文档）：
   - 从异常体系提取需覆盖的异常类型
   - 从接口参数校验规则提取边界值
   - 从分层架构提取各层测试策略

5. **进入测试生成流程**

## 任务模式

| 用户意图 | 模式 | 动作 |

| 不明确 | **询问** | 询问用户具体需求 |

## 测试生成流程

### 分步输出策略

| 阶段 | 输出内容 | 暂停点 |
|------|----------|--------|
| 阶段一 | 分析被测类，输出测试策略表（方法 × 场景矩阵） | 确认测试范围和场景覆盖 |
| 阶段二 | 生成测试代码 | 确认代码无误后执行 |
| 阶段三 | 执行测试 → 分析失败 → 修复 → 重新执行 | 直到全部通过 |
| 阶段四 | 输出测试报告（覆盖率矩阵 + 执行结果） | 确认覆盖率达标 |

### 用户交互路线图

```
前置 ──→ [询问] 确认被测目标 + 技术栈
  │
阶段一 ──→ [暂停] 确认测试策略表（方法 × 场景矩阵）
  │
阶段二 ──→ [暂停] 确认生成的测试代码
  │
阶段三 ──→ [循环] 执行 → 分析失败 → 修复 → 重新执行（直到全部通过）
  │
阶段四 ──→ 输出测试报告
```

### 测试策略表模板

| 被测方法 | 正常路径 | 边界条件 | 异常路径 | 备注 |
|---------|---------|---------|---------|------|
| `createXxx(dto)` | 参数合法，成功创建 | 名称为空字符串；字段达到最大长度 | 名称重复抛 DuplicateKeyException | — |
| `getById(id)` | id存在，返回VO | id=0 | id不存在抛 ResourceNotFoundException | — |
| `deleteById(id)` | id存在，软删除 | — | id不存在抛异常 | — |

> **AI生成规则**：
> - 若有详细设计文档，从异常体系（lyspec-backend 5.2节）提取异常路径
> - 从接口参数校验规则（lyspec-backend 3.4节）提取边界值
> - 每个 public 方法至少覆盖正常路径 + 一个异常路径

## 按层测试策略（后端 Java）

> 从 lyspec-backend 分层架构（4.2节）推导各层测试重点。

| 层次 | 测试重点 | Mock 对象 | 不测内容 |
|------|---------|----------|---------|
| Controller | 参数校验、权限检查、响应格式 | Service | 业务逻辑（由Service层测试覆盖） |
| Service | 业务逻辑、事务边界、异常处理 | Mapper、外部Service | SQL执行细节 |
| Converter | DTO↔Entity↔VO 转换正确性 | 无 | — |
| 工具类 | 纯函数输入输出 | 无 | — |

> **AI生成规则**：
> - 优先覆盖 Service 层（业务逻辑集中区）
> - Controller 层仅测试参数校验和异常处理，不测试业务逻辑
> - Mapper/Repository 层不做单元测试（由集成测试覆盖）

---

## 按类型测试策略（前端）

> 从 lyspec-frontend 组件设计（第三章）和状态管理（第五章）推导测试重点。

| 类型 | 测试重点 | Mock 对象 | 工具 |
|------|---------|----------|------|
| 工具函数 | 纯函数输入输出、边界值 | 无 | Vitest |
| Composable/Hook | 响应式数据变化、生命周期 | API层 | Vitest |
| Pinia Store | action 执行、state 变化、getter 计算 | API层 | Vitest + createTestingPinia |
| Vue 组件 | props渲染、事件触发、插槽、条件渲染 | Store、API | @vue/test-utils |
| API 层 | 请求参数拼装、响应解析 | HTTP客户端 | Vitest |

> **AI生成规则**：
> - 优先覆盖工具函数和 Composable（纯逻辑，测试价值最高）
> - 组件测试关注行为而非实现细节，不测试 DOM 结构
> - API 层验证请求参数正确性，不验证 HTTP 库内部行为

## 附录

### A. 用户交互检查点

- [ ] 被测目标已确认（文件路径/模块名）
- [ ] 技术栈已确认（自动识别或用户指定）
- [ ] 阶段一完成：测试策略表已确认（方法 × 场景矩阵）
- [ ] 阶段二完成：测试代码已确认
- [ ] 阶段三完成：所有测试通过
- [ ] 阶段四完成：覆盖率报告已输出，达标 ≥ 80%

### B. 质量检查点

**测试充分性**：
- [ ] 每个 public 方法至少有一个正常路径测试
- [ ] 有条件分支的方法覆盖了主要分支
- [ ] 异常路径覆盖了详细设计中定义的所有异常类型
- [ ] 边界值覆盖了详细设计中定义的参数校验规则

**测试质量**：
- [ ] 命名规范：Java 使用 `should_xxx_when_yyy`，前端使用中文描述
- [ ] 每个测试只验证一个行为
- [ ] 无跳过的测试（`@Disabled` / `it.skip`）
- [ ] 无状态共享（每个测试独立）
- [ ] Mock 合理：只 Mock 外部依赖，不 Mock 被测对象

**隔离性**：
- [ ] 不依赖外部环境（数据库、网络、文件系统）
- [ ] 不依赖测试执行顺序
- [ ] 可重复执行，结果稳定
