# LY 团队技术选型规范

> 最近审查日期：2026-03-17

## 核心原则

1. 所有技术选型必须在本清单范围内
2. 版本号必须符合清单中定义的版本约束
3. 引入清单外的技术须提交团队技术评审，评审通过后方可使用
4. AI 不得推荐或生成使用清单外技术的代码或方案

## 数据库

### 关系型数据库

> **默认推荐规则**：
> - **通用项目**（无国产化要求）：默认使用 **PostgreSQL**。理由：功能丰富（JSONB、全文检索、PostGIS）、生态成熟、与 KingbaseES SQL 兼容度高，便于后续国产化迁移。
> - **国产化项目**（有信创/国产化要求）：默认使用 **KingbaseES**。理由：基于 PostgreSQL 内核，SQL 兼容性好；通过国产化认证；从 PostgreSQL 迁移成本极低。

## 使用规则

1. **必须遵守**：所有技术选型必须在本清单范围内，不得绕过
2. **版本约束**：版本号必须符合清单中的版本范围，不得使用过低或过高的未经评审版本
3. **默认选型**：用户未明确指定时，按"默认推荐"标注执行——通用项目默认 PostgreSQL，国产化项目默认 KingbaseES
4. **国产化优先**：涉及国产化/信创项目时，优先选择国产化标注的技术（KingbaseES、银河麒麟、统信 UOS 等）
5. **清单外技术**：如业务场景确实需要清单外技术，须提交技术评审申请，经团队评审通过后更新本清单
6. **版本审查周期**：本清单每季度审查一次，检查是否有安全漏洞、EOL 风险或重大版本升级
