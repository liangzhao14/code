# 详细设计文档 — 全栈编排器（AI友好版）

> 本文件是全栈详细设计的**编排层**，不包含具体章节模板。
> 具体前端设计模板见 `lyspec-frontend/lyspec-frontend.md`，后端设计模板见 `lyspec-backend/lyspec-backend.md`。
> 本文档处于软件生命周期的"架构设计→**详细设计**→代码编写"阶段，承上启下。

---

## 定位与职责

```
lyspec-detail（本文件 — 编排器）
├── lyspec-frontend  ← 前端详细设计模板（页面路由、组件树、TypeScript类型、API封装、性能、部署）
└── lyspec-backend   ← 后端详细设计模板（接口设计、分层架构、异常体系、事务、数据设计、监控）
```

| 使用场景 | 推荐 skill |
|---------|-----------|
| 全栈模块（前端+后端+数据库） | **lyspec-detail**（本文件）→ 编排两个子 skill，合并为一份文档 |
| 纯前端项目 | 直接使用 **lyspec-frontend** |
| 纯后端服务 | 直接使用 **lyspec-backend** |

## 第一步：确认上游文档

> 详细设计的所有内容均以架构设计文档和PRD为核心输入。

### 上游依赖文档

| 文档 | 新项目/大版本升级 | 小功能迭代 | 提取内容 |
|------|:----------------:|:----------:|----------|
| 产品需求规格说明书（PRD） | **必需** | **必需** | 功能需求、业务流程、页面原型、交互规则 |
| 架构设计文档（lyspec-arch产物） | **必需** | 可选 | 模块划分、技术选型、物理架构、系统流程、非功能需求 |
| 功能规格说明书 | 可选 | 可选 | 功能点细化、业务规则、异常流程 |
| UI/UE设计产物（lyspec-ued产物） | 可选 | 可选 | HTML Demo、页面清单表、组件树 |

> **判断规则**：
> - **新项目/大版本升级**：首次开发、技术栈变更、系统重构、新增子系统 → 架构设计文档**必需**
> - **小功能迭代**：在已有系统上新增功能页面、修改业务逻辑、增加接口 → 架构设计文档**可选**（技术栈和架构沿用既有方案）
> - AI须在开始时询问用户："这是新项目/大版本升级，还是在已有系统上的小功能迭代？" 据此决定架构设计文档是否必需

### AI执行流程

1. 若用户已指定文档路径，直接读取；若未指定，询问：
   - "这是新项目/大版本升级，还是小功能迭代？"
   - "请提供PRD文档路径（必需）"
   - 新项目/大版本升级："请提供架构设计文档路径（必需）"
   - 小功能迭代："是否有架构设计文档？（可选，无则沿用既有技术栈和架构）"
   - "是否有功能规格说明书或UI/UE设计产物？（可选）"

2. 从上游文档提取关键信息：

   **从PRD提取**（始终执行）：
   - 本模块的功能需求清单
   - 页面/交互设计要求
   - 业务规则和约束条件
   - 角色权限要求

   **从架构设计文档提取**（新项目/大版本升级时执行；小功能迭代若有则提取，无则沿用既有架构）：
   - 本模块在整体架构中的定位（第12节组件定义表）
   - 本模块的技术选型（第15节物理架构、第19节技术选型清单）
   - 本模块涉及的系统流程（第13节）
   - 非功能性需求（第10节）
   - 与其他模块的依赖关系（第12节依赖关系列）

   > **小功能迭代无架构设计文档时**：技术栈、架构模式、部署方式沿用既有系统，AI从已有代码仓库或用户描述中推导，并标注"沿用既有架构"。

## 全栈文档编排流程

> 以下流程仅在**全栈模块**的**文档生成模式**下执行。纯前端/纯后端直接使用对应子 skill。

### 生成模式选择

| 模式 | 说明 | 适用场景 |
|------|------|----------|
| **逐步确认模式**（默认） | 分4个阶段输出，每阶段暂停等待用户确认 | 核心模块、首次使用、需求不明确 |
| **一次性生成模式** | AI一次输出全部内容，最后统一确认 | 简单CRUD模块、需求清晰 |

### 分步输出策略（后端先行，围绕接口编程）

> **设计理念**：后端先定义API契约（接口URL、参数、错误码、响应结构），前端基于契约生成TypeScript类型、对接接口。
> 这样前后端一致性天然有保障，减少返工。

| 阶段 | 输出内容 | 来源 | 暂停点 |
|------|----------|------|--------|
| 阶段一 | 概览 + 设计考虑 | lyspec-backend 第一~二章 | 确认背景理解和设计约束 |
| 阶段二 | 后端设计（接口+内部+外部依赖） | **调用 lyspec-backend** 第三~七章 | ★ 确认接口设计、错误码、类图、核心流程 |
| 阶段三 | 数据设计 + 部署运维 | **调用 lyspec-backend** 第八~九章 | 确认表结构、部署方案 |
| 阶段四 | 前端详细设计 | **调用 lyspec-frontend** 第二~八章 | 确认页面拆分、组件设计、接口对接 |

> **阶段四说明**：前端设计基于阶段二已确认的后端接口契约，自动生成：
> - TypeScript 类型定义（从后端参数表/响应体推导）
> - 接口对接清单（直接引用后端接口URL和Method）
> - 错误处理设计（从后端错误码表推导前端处理策略）

### 用户交互路线图

```
前置 ──→ [询问] 确认上游文档 + 技术栈 + 模块类型
  │
阶段一 ──→ [暂停] 确认背景和设计约束
  │
阶段二 ──→ [暂停] ★ 确认后端接口设计 + 错误码 + 类图（调用 lyspec-backend）
  │                    ↓ 接口契约确定
阶段三 ──→ [暂停] 确认数据设计 + 部署方案
  │                    ↓ 后端设计完成
阶段四 ──→ [暂停] 确认前端设计（基于后端契约，调用 lyspec-frontend）
  │
完成 ──→ 前后端一致性校验 ──→ 输出合并文档 + 下游产物
```

### 全栈文档合并策略

全栈模块的最终产物为一份合并文档，按**后端先行**顺序组织章节：

```
{模块名}_详细设计_{日期}.md
├── 一、概览                     ← lyspec-backend 第一章
├── 二、设计考虑                  ← lyspec-backend 第二章
├── 三、后端接口设计（API契约）    ← lyspec-backend 第三章（全文核心，前端依赖此章）
├── 四、内部设计                  ← lyspec-backend 第四~六章内容合并
│   ├── 4.1 分层架构
│   ├── 4.2 核心类图
│   ├── 4.3 关键时序图
│   ├── 4.4 核心流程图
│   ├── 4.5 异常处理体系
│   ├── 4.6 事务设计
│   └── 4.7 高可用与安全设计
├── 五、外部依赖                  ← lyspec-backend 第七章
├── 六、数据设计                  ← lyspec-backend 第八章
├── 七、前端详细设计              ← lyspec-frontend 第二~七章内容合并（基于第三章API契约）
│   ├── 7.1 页面与路由设计
│   ├── 7.2 组件树设计
│   ├── 7.3 状态管理设计
│   ├── 7.4 前端接口对接清单      ← 直接引用第三章接口定义
│   ├── 7.5 前端权限控制设计
│   ├── 7.6 关键交互流程
│   └── 7.7 前端错误处理设计      ← 从第三章错误码推导
└── 八、部署与运维                ← lyspec-backend 第九章 + lyspec-frontend 第八章合并
```

> **文档产物命名规则**（遵循 lyspec-orchestrator 统一命名规范）：
> - 若用户指定了文档名称，使用用户指定的名称
> - 若用户未指定，按 `{projectName}_{projectVersion}_DETAIL_{YYYY-MM-DD}.md` 格式命名
> - 存放到 `doc/` 目录

## 前后端一致性校验（独有价值）

> 这是全栈编排器的核心增量能力。子 skill 各自不具备跨端校验能力。

全栈文档合并完成后，**必须执行**以下校验：

### 接口一致性

| 校验项 | 前端来源 | 后端来源 | 校验规则 |
|--------|---------|---------|---------|
| 接口 URL | lyspec-frontend 4.3节 | lyspec-backend 3.3节 | URL + Method 完全一致 |
| 请求参数字段 | lyspec-frontend 4.2节 TypeScript 类型 | lyspec-backend 3.4节参数表 | 字段名、类型、必填一致 |
| 响应体结构 | lyspec-frontend 4.1节 Result 类型 | lyspec-backend 3.1节响应体格式 | code/message/data 结构一致 |
| 分页参数 | lyspec-frontend QueryDTO | lyspec-backend 分页接口 | page/size/total 字段名一致 |

### 错误处理一致性

| 校验项 | 前端来源 | 后端来源 | 校验规则 |
|--------|---------|---------|---------|
| HTTP状态码处理 | lyspec-frontend 7.1节 | lyspec-backend 3.2节 | 前端处理的状态码须覆盖后端定义的所有状态码 |
| 业务错误码 | lyspec-frontend 错误提示 | lyspec-backend 错误码表 | 前端有对应的用户提示 |
| 认证失败 | lyspec-frontend 401处理 | lyspec-backend 认证方式 | JWT/OAuth2 机制匹配 |

### 权限一致性

| 校验项 | 前端来源 | 后端来源 | 校验规则 |
|--------|---------|---------|---------|
| 角色定义 | lyspec-frontend 7.3节 | lyspec-backend 6.2节 | 角色名称完全一致 |
| 路由权限 | lyspec-frontend 2.1节 meta.roles | lyspec-backend 接口鉴权 | 前端路由守卫与后端鉴权匹配 |
| 按钮权限 | lyspec-frontend 7.3节 | lyspec-backend 数据权限 | 前端权限点与后端权限标识对应 |

> **AI生成规则**：
> - 校验不通过时，列出差异清单并建议修改方案
> - 优先以后端接口定义为准，前端对齐后端
> - 校验结果附加在文档末尾的"质量检查报告"中
